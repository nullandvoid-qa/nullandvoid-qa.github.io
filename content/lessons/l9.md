<h2>Critérios de Aceitação e Definição de Pronto</h2>

  <h3>🎯 Objetivos de Aprendizado</h3>
  <p>Após esta aula, você será capaz de:</p>
  <ul>
    <li>Entender a importância dos critérios de aceitação</li>
    <li>Escrever critérios claros, completos e testáveis</li>
    <li>Conectar aceitação de histórias com a Definição de Pronto</li>
    <li>Avaliar se uma história está realmente pronta para ser entregue</li>
  </ul>

  <h3>📊 Resumo Executivo</h3>
  <p>Critérios de aceitação não são apenas uma formalidade. Eles são a linguagem que transforma expectativas do cliente em testes reais. Uma história sem critérios bem definidos é um convite ao retrabalho e à interpretação errada.</p>

  <h3>🧩 O que é Critério de Aceitação?</h3>
  <p>É uma condição que a história precisa satisfazer para ser considerada concluída. Devem ser:</p>
  <ul style="margin:1rem 0">
    <li><strong>Claros</strong> — qualquer pessoa deve entender o resultado esperado</li>
    <li><strong>Testáveis</strong> — não podem ser ambíguos ou genéricos</li>
    <li><strong>Completos</strong> — cobrem comportamento normal e exceções importantes</li>
  </ul>

  <h3>📌 Exemplo</h3>
  <p>História: “Como usuário, quero cadastrar meu cartão para pagar compras”.</p>
  <ul>
    <li>O usuário deve ver uma mensagem de confirmação ao salvar o cartão</li>
    <li>Se o número do cartão for inválido, o sistema deve mostrar erro específico</li>
    <li>A data de validade deve ser aceita apenas em formato MM/AA</li>
  </ul>

<div style="background:#fef3c7; padding:1.5rem; border-radius:0.5rem; margin:1.5rem 0; border-left:4px solid #f59e0b">
<p><strong>Antes vs Depois: Critérios Mal Escritos vs Bem Escritos</strong></p>

<p><strong>❌ Critérios MAL Escritos (Vagos, Não Testáveis):</strong></p>
<ul style="margin:1rem 0">
  <li>"O sistema deve funcionar bem" - <em>O que significa "funcionar bem"?</em></li>
  <li>"A interface deve ser bonita" - <em>Subjetivo, não verificável</em></li>
  <li>"O usuário deve conseguir fazer login" - <em>Como? Em quanto tempo? Com quais dados?</em></li>
  <li>"O pagamento deve ser seguro" - <em>Seguro como? HTTPS? Criptografia? Validação?</em></li>
  <li>"O sistema deve ser rápido" - <em>Rápido quanto? 1s? 10s? Em qual dispositivo?</em></li>
</ul>

<p><strong>✅ Critérios BEM Escritos (Claros, Testáveis, Mensuráveis):</strong></p>
<ul style="margin:1rem 0">
  <li>"O usuário deve conseguir fazer login em menos de 3 segundos usando email e senha válidos" - <em>Tempo específico, dados específicos</em></li>
  <li>"A interface deve seguir o guia de estilo da empresa e passar no teste de acessibilidade WCAG AA" - <em>Referência clara, verificável</em></li>
  <li>"O pagamento deve usar HTTPS com TLS 1.3 e não armazenar dados do cartão no servidor" - <em>Tecnologia específica, comportamento claro</em></li>
  <li>"O formulário de cadastro deve validar email em formato correto e mostrar erro específico se inválido" - <em>Comportamento específico, feedback claro</em></li>
  <li>"A página de checkout deve carregar em menos de 2 segundos em 4G" - <em>Tempo específico, condição específica</em></li>
</ul>
</div>

<div style="background:#f0fdf4; padding:1.5rem; border-radius:0.5rem; margin:1.5rem 0; border-left:4px solid #10b981">
<p><strong>Exemplo Prático: Feature de Reset de Senha</strong></p>

<p><strong>❌ Versão Ruim:</strong></p>
<ul style="margin:1rem 0">
  <li>"O usuário deve conseguir resetar a senha"</li>
  <li>"O sistema deve enviar um email"</li>
  <li>"O link deve funcionar"</li>
</ul>

<p><strong>✅ Versão Boa:</strong></p>
<ul style="margin:1rem 0">
  <li>"Ao solicitar reset, o usuário deve receber email em até 2 minutos com link válido por 24 horas"</li>
  <li>"O link deve redirecionar para página de redefinição com token pré-preenchido"</li>
  <li>"Nova senha deve ter mínimo 8 caracteres, incluindo número e caractere especial"</li>
  <li>"Após redefinição bem-sucedida, o usuário deve ser redirecionado para login com mensagem de sucesso"</li>
  <li>"Se link for usado mais de uma vez, sistema deve mostrar erro de 'link expirado'"</li>
  <li>"Sistema deve bloquear solicitações múltiplas do mesmo IP em menos de 5 minutos (rate limiting)"</li>
</ul>
</div>

<div style="background:#dbeafe; padding:1.5rem; border-radius:0.5rem; margin:1.5rem 0; border-left:4px solid #3b82f6">
<p><strong>Checklist para Avaliar Critérios de Aceitação:</strong></p>
<ul style="margin:1rem 0">
  <li>✅ <strong>É específico?</strong> - Não usa termos vagos como "bom", "rápido", "fácil"</li>
  <li>✅ <strong>É testável?</strong> - Posso escrever um teste que valide isso?</li>
  <li>✅ <strong>É mensurável?</strong> - Tem números, tempos, quantidades específicas?</li>
  <li>✅ <strong>É completo?</strong> - Cobre o comportamento normal e casos de erro importantes?</li>
  <li>✅ <strong>É atômico?</strong> - Cada critério testa uma coisa só, não múltiplas coisas juntas?</li>
  <li>✅ <strong>É necessário?</strong> - Se este critério falhar, a feature não entrega valor?</li>
</ul>
</div>

  <h3>✅ Definição de Pronto (Definition of Done)</h3>
  <p>A Definição de Pronto é a checklist que garante que o trabalho está finalizado. Não é apenas o código entregue: é o comportamento validado, a documentação atualizada e a qualidade confirmada.</p>

  <h3>🛠️ Exemplo de itens de DoD</h3>
  <ul style="margin:1rem 0">
    <li>Código revisado e aprovado</li>
    <li>Testes automatizados adicionados para comportamento crítico</li>
    <li>Critérios de aceitação validados manualmente</li>
    <li>Documentação de uso atualizada quando necessário</li>
    <li>Bugfixes verificados e regressão executada</li>
  </ul>

  <h3>📋 Templates de Critérios de Aceitação por Tipo de Projeto</h3>
  <div style="background:#f0fdf4; padding:1.5rem; border-radius:0.5rem; margin:1.5rem 0; border-left:4px solid #10b981">
  <p><strong>Projeto E-commerce:</strong></p>
  <ul style="margin:1rem 0">
    <li>✓ Usuário consegue completar checkout em < 3 minutos</li>
    <li>✓ Sistema calcula corretamente impostos e frete</li>
    <li>✓ Pagamento é processado com confirmação clara</li>
    <li>✓ Usuário recebe email de confirmação</li>
    <li>✓ Estoque é atualizado após compra confirmada</li>
  </ul>

  <p><strong>Projeto Mobile App:</strong></p>
  <ul style="margin:1rem 0">
    <li>✓ App funciona offline para funcionalidades críticas</li>
    <li>✓ Sincronização de dados funciona quando conexão retorna</li>
    <li>✓ Notificações push são recebidas corretamente</li>
    <li>✓ App não crasha em memória baixa</li>
    <li>✓ Interface responde em < 200ms em dispositivos médios</li>
  </ul>

  <p><strong>Projeto API/B2B:</strong></p>
  <ul style="margin:1rem 0">
    <li>✓ API responde em < 100ms para 95% das requisições</li>
    <li>✓ Rate limiting funciona corretamente</li>
    <li>✓ Autenticação é validada em todos os endpoints</li>
    <li>✓ Logs contêm informações suficientes para debug</li>
    <li>✓ Documentação da API está atualizada</li>
  </ul>
  </div>

  <h3>✏️ Exercício Prático</h3>
  <p>Escolha uma história simples e escreva três critérios de aceitação e três itens de Definição de Pronto. Compare com seu time e veja se sobrou algo ambiguo.</p>

  <h3>🧠 Regra de Ouro</h3>
  <p>Se um critério de aceitação não permite verificar se a história foi entregue, ele precisa ser reescrito. Boa aceitação é uma promessa de comportamento, não um resumo vago.</p>
