<h2>Definition of Done e Qualidade em Agile</h2>

<h3>🎯 Objetivos de Aprendizado</h3>
<p>Após esta aula, você será capaz de:</p>
<ul>
  <li>Entender o papel do Definition of Done no processo Scrum</li>
  <li>Conectar DoD com qualidade e responsabilidade do time</li>
  <li>Escrever itens de DoD que realmente aumentam confiança</li>
  <li>Aplicar DoD para evitar entregas incompletas ou mal testadas</li>
</ul>

<h3>📊 Resumo Executivo</h3>
<p>Definition of Done é a lista mínima que garante que uma história não só foi concluída, mas também testada, revisada e pronta para ser entregue. Ela estabelece um padrão comum de qualidade para o time.</p>

<h3>📌 Itens comuns de DoD</h3>
<ul style="margin:1rem 0">
  <li>Código revisado e integrado</li>
  <li>Testes automatizados ou manuais executados</li>
  <li>Critérios de aceitação atendidos</li>
  <li>Documentação atualizada se necessário</li>
  <li>Critérios de segurança e performance verificados</li>
</ul>

<h3>📋 Exemplos de DoD Reais por Tipo de Projeto</h3>
<div style="background:#f0fdf4; padding:1.5rem; border-radius:0.5rem; margin:1.5rem 0; border-left:4px solid #10b981">
<p><strong>DoD para Projeto Financeiro:</strong></p>
<ul style="margin:1rem 0">
  <li>✓ Código passou por review de segurança</li>
  <li>✓ Testes de cobertura de branches > 80%</li>
  <li>✓ Testes de performance validados (SLA < 200ms)</li>
  <li>✓ Auditoria de logs sensíveis realizada</li>
  <li>✓ Testes de penetração aprovados</li>
  <li>✓ Documentação de compliance atualizada</li>
</ul>

<p><strong>DoD para Projeto E-commerce:</strong></p>
<ul style="margin:1rem 0">
  <li>✓ Testes E2E de checkout passando</li>
  <li>✓ Validação de mobile responsiveness</li>
  <li>✓ Testes de acessibilidade (WCAG 2.1 AA)</li>
  <li>✓ Performance (Lighthouse score > 90)</li>
  <li>✓ Testes de cross-browser (Chrome, Firefox, Safari)</li>
  <li>✓ Monitoramento de analytics configurado</li>
</ul>

<p><strong>DoD para Projeto de API:</strong></p>
<ul style="margin:1rem 0">
  <li>✓ Documentação da API atualizada (Swagger/OpenAPI)</li>
  <li>✓ Testes de contrato (consumer-driven)</li>
  <li>✓ Testes de carga validados</li>
  <li>✓ Versionamento de API documented</li>
  <li>✓ Logs estruturados implementados</li>
  <li>✓ Rate limiting testado e documentado</li>
</ul>
</div>

<h3>📊 Métricas de Qualidade para DoD</h3>
<div style="background:#dbeafe; padding:1.5rem; border-radius:0.5rem; margin:1.5rem 0; border-left:4px solid #3b82f6">
<p><strong>Métricas Quantitativas:</strong></p>
<ul style="margin:1rem 0">
  <li>📈 Cobertura de testes > 80%</li>
  <li>🐛 Bug escape rate < 5%</li>
  <li>⚡ Tempo de execução de testes < 15min</li>
  <li>🔄 Taxa de retrabalho < 10%</li>
</ul>

<p><strong>Métricas Qualitativas:</strong></p>
<ul style="margin:1rem 0">
  <li>✅ Critérios de aceitação claros e testáveis</li>
  <li>🎯 Requisitos bem definidos e sem ambiguidade</li>
  <li>🧪 Testes cobrem casos principais e bordas</li>
  <li>📚 Documentação atualizada e acessível</li>
</ul>
</div>

<h3>🧠 Papel do QA sênior</h3>
<p>O QA sênior ajuda a transformar DoD em medidas práticas e evita itens vagos. Não basta ter “testes feitos”; é preciso saber quais testes entregam valor real.</p>

<h3>⚠️ Anti-padrões</h3>
<ul style="margin:1rem 0">
  <li>DoD muito curto: “código pronto” sem validação</li>
  <li>DoD muito longo: checklist inatingível que atrasa entrega</li>
  <li>Itens que não são verificáveis ou não agregam qualidade</li>
</ul>

<h3>✏️ Exercício Prático</h3>
<p>Escreva cinco itens de Definition of Done para uma história de “recuperar senha” e explique por que cada item protege a qualidade.</p>

<h3>🧠 Regra de Ouro</h3>
<p>Um DoD ruim dá a sensação de que a entrega terminou; um DoD bom cria confiança real de que a entrega foi feita com qualidade, segurança e clareza.</p>

<h3>✅ Fechamento</h3>
<p>Ao final desta sequência, você já consegue pensar em qualidade como um sistema completo: requisitos claros, testes bem priorizados, automação inteligente e um padrão de entrega que reduz retrabalho e aumenta confiança.</p>
